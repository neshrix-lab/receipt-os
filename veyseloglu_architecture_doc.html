01

Концепция · почему остатки центральные

В классической системе приёмки фокус — на накладной. В Receipt OS фокус сместился: остатки определяют всё остальное. Из остатков рождаются заказы, заказы порождают приёмки, приёмки фиксируют расхождения, расхождения становятся претензиями. Понимание остатков по всей сети — единственный способ дать менеджеру и агентам полную картину.

🔑
Ключевой принцип
Receipt OS не хранит остатки. Источник истины — 1С УТ. Receipt OS читает остатки в реальном времени, обрабатывает агентами, визуализирует, и принимает решения на их основе.

Источники данных по остаткам

Остатки в холдинге Veysəloğlu разбросаны по трём уровням. Каждый уровень — свой источник данных и свой способ синхронизации.

Уровень 1 · Центральный склад

Главный распределительный хаб Veysəloğlu. Сюда приходят все поставки от внешних поставщиков (Parvaз, Coca-Cola, ProFood, Nestlé...). Отсюда товар уходит в магазины. Receipt OS читает остатки SELECT * FROM Регистр.ОстаткиТоваров через COM или HTTP-сервис 1С.

Уровень 2 · Магазины сети

Сеть состоит из нескольких брендов: OBA Market, Araz Market, Rossmann и другие. У каждого бренда — десятки или сотни точек. Каждая точка ведёт свой учёт в 1С. Receipt OS агрегирует остатки по магазинам через единый интеграционный слой 1С холдинга.

Уровень 3 · В пути

Это товары которые уже заказаны, но физически ещё не на полке. Включает: открытые заказы поставщикам (ожидаем поступления на центральный склад) и внутренние перемещения (отгружено с центра, не принято в магазине). Receipt OS читает эти данные из 1С чтобы прогнозы учитывали будущие поступления.

03

Граница с 1С УТ и магазинами

Receipt OS не дублирует 1С. Чёткое разделение ответственности: 1С хранит истину, Receipt OS делает её доступной для агентов и людей. Магазины сети — автономные единицы со своими системами. Receipt OS взаимодействует с ними только через документы 1С холдинга.

Важно для архитектуры
Receipt OS не пишет остатки в 1С напрямую. Остатки пересчитываются 1С автоматически на основе документов которые система создала (приходных накладных, перемещений). Это сохраняет 1С как единственный источник истины.
❓
Критичный вопрос для уточнения у клиента
Магазины сети работают на едином холдинговом 1С (одна база, разные организации) или у каждого магазина/бренда свой отдельный экземпляр 1С? От ответа зависит сложность интеграции и скорость синхронизации данных. Единый холдинговый 1С — интеграция тривиальная. Отдельные экземпляры — нужен интеграционный слой между ними.

05

Шесть агентов системы

Каждый агент решает одну задачу. Вместе они закрывают полный цикл управления остатками и закупками. Не каждый «агент» использует LLM — некоторые это просто детерминированный код.

🔄
Агент-агрегатор
Код

Собирает остатки со всех 1С (центральный + магазины), кэширует в Supabase, обеспечивает быстрый доступ для других агентов и UI.

1С УТ → COM/HTTP → Supabase pgvector → API для агентов
🔮
Агент-прогноз потребления
AI

Смотрит на историю продаж за 90 дней + сезонность + текущий остаток → предсказывает когда закончится товар на складе или в магазине.

История продаж + остаток → ML-модель → дата дефицита по SKU
📋
Агент-планировщик заказов
AI

Объединяет прогноз дефицита + время доставки поставщика + рейтинг поставщика → предлагает что и когда заказывать. Учитывает альтернативных поставщиков.

Прогноз + рейтинги + сроки → Claude → рекомендация заказа
🚚
Агент-распределитель
AI

Видит остатки на складе и потребности по 500-2000 магазинов → предлагает оптимальное распределение. Находит переполнения и недостачи, балансирует.

Остатки склад + остатки магазины → линейное программирование + LLM → план отгрузок
⚠️
Агент-аномалист
AI

Мониторит паттерны: систематические недопоставки, регулярные расхождения цен, странные потери в пути «склад → магазин». Эскалирует в Inbox.

История приёмок + статистика → Claude интерпретация → алерт менеджеру
📨
Агент-переговорщик
AI

Драфтит претензии, письма поставщикам, запросы на корректировки. Менеджер только проверяет и подписывает — экономия часов работы.

Данные о расхождении → Claude генерация текста → менеджер подпись → Email/портал
🤖
Координация агентов
Агенты работают независимо, но обмениваются результатами через общую базу. Каждый может «спросить» другого: планировщик читает прогноз, аномалист видит план заказов, переговорщик использует данные аномалиста.

📥 Inbox · очередь решений

Что меняется: задачи в Inbox приходят не только от поставщиков, но и от магазинов — через документы 1С. Заявка магазина на пополнение → агент-планировщик сверяет с остатками склада → если нужно заказывать у поставщика, появляется задача в Inbox. Акт расхождений от магазина → агент-аномалист анализирует паттерн → алерт «систематические потери на маршруте Склад → Araz Xətai».

Источник: 1С магазина → XML/HTTP обмен → 1С холдинга → агент-агрегатор → Inbox.

📦 Остатки · три уровня

Что меняется: раздел теперь показывает три слоя. Центральный склад — в реальном времени через COM. Магазины — агрегированно, с задержкой до нескольких часов (зависит от частоты синхронизации 1С). «В пути» — заказы отгруженные но ещё не принятые магазином.

Важно: данные по магазинам — это агрегат, не детальный учёт. Receipt OS видит «в OBA Nizami осталось 48 ед. масла», а не «масло на полке номер 3». WMS магазина управляет деталями.

🏢 Поставщики · 4-параметровый рейтинг

Что меняется: рейтинг учитывает точность цен, количество, сроки и качество. Данные по количеству и качеству берутся из актов приёмки на центральном складе — не из магазинов. Магазины не оценивают поставщиков напрямую, они принимают товар уже от склада Veysəloğlu.

Источник: 1С УТ (заказы + накладные + акты приёмки) → агент-аномалист → пересчёт рейтинга раз в день.

📋 Заказы · два потока

Внешние заказы (Veysəloğlu → поставщикам) — формирует агент-планировщик на основе остатков склада и прогноза потребления по всей сети.

Внутренние заявки (магазин → центральный склад) — создаются в 1С магазина, Receipt OS читает их и показывает менеджеру. Агент-распределитель предлагает оптимальное распределение между магазинами — иногда лучше перебросить из другого магазина, чем заказывать у поставщика.

📄 Рабочее место оператора

Что меняется: при приёмке товара на центральный склад оператор видит контекст — какие магазины ждут эти позиции, с какой срочностью. Если приёмка показала недопоставку — агент сразу пересчитывает сколько магазинов не получат плановый товар.

Источник: заявки магазинов из 1С → агент-агрегатор кэширует → показывает оператору при открытии накладной.

📤 Портал поставщика

Что меняется: поставщик видит агрегированный прогноз потребления по своим SKU — рассчитанный на основе продаж по всей сети магазинов. Это даёт поставщику возможность планировать производство и предлагать выгодные условия заранее.

Важно: поставщик видит только прогноз по своим позициям и только агрегированные цифры — без разбивки по конкретным магазинам.

📈 Аналитика · внутренние потери

Новый блок которого нет в классических системах: потери при внутренних перемещениях «склад → магазин». Receipt OS сравнивает расходную накладную (что отгружено) с приходной накладной магазина (что принято). Систематические расхождения по конкретному маршруту или магазину — сигнал для службы безопасности.

Источник: расходные накладные 1С холдинга + приходные накладные 1С магазинов → агент-аномалист → раздел Аналитика.

Список того что Receipt OS сознательно оставляет в 1С или вне зоны ответственности. Это важно для разговора с клиентом — чтобы не было ложных ожиданий.

Не ведём учёт остатков1С УТИсточник истины должен быть один. Дублирование = расхождения и ошибки.
Не делаем физическую приёмку1С Склад + сканерыСканеры штрихкодов работают в 1С. Мы читаем результат.
